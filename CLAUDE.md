## What this is

`uniclip` is an end-to-end-encrypted "universal clipboard": a web SPA syncs clipboard text and files between devices, end-to-end encrypted, through a relay that **never sees plaintext or keys**. pnpm + Turborepo monorepo, TypeScript everywhere. Current capability is E2EE text + chunked binary file transfer + synced delete; the **WebRTC fast path** (Phase 3 / v0.3) lets content travel peer-to-peer (LAN-direct) with the relay as signaling + content fallback. The earlier "v0.1 text-only" framing is historical.

## Commands

```bash
pnpm install                         # bootstrap the workspace
pnpm typecheck                       # turbo: tsc --noEmit (svelte-check for web) across all packages
pnpm test                            # turbo: unit tests for all packages EXCEPT e2e
pnpm test:e2e                        # Playwright two-browser test (boots relay + web dev servers itself)

# single package / single test file (vitest name pattern)
pnpm --filter @uniclip/crypto test envelope
pnpm --filter @uniclip/relay test rooms     # same name-pattern arg, routed through Bun
pnpm --filter @uniclip/relay typecheck

# local dev (split origin: web on :5173 talks to relay on :3000)
PORT=3000 pnpm --filter @uniclip/relay dev          # bun run --hot
VITE_RELAY_BASE=http://localhost:3000 pnpm --filter @uniclip/web dev

# production image: one container serves API + SPA on :3000
docker build -t uniclip:dev . && docker run --rm -p 3000:3000 uniclip:dev

# test across real devices: clipboard needs a secure context (HTTPS or localhost),
# so http://<lan-ip> won't sync. Expose the container over HTTPS with Tailscale:
tailscale serve --bg 3000      # or --https=8443 3000 if :443 is taken on the host
```

**`pnpm test` deliberately excludes `@uniclip/e2e`** (`--filter=!@uniclip/e2e`): Playwright must run after `playwright install`, and booting its dev servers starves the unit suite's async-timing tests. Run e2e only via `pnpm test:e2e`. The `@uniclip/e2e` package lives in the **top-level `e2e/` dir** (a standalone workspace entry beside `apps/*` and `packages/*`), not under either of them.

## Architecture

Data flow: **web SPA ⇄ relay (WebSocket) ⇄ web SPA**. The relay is a pure in-memory fan-out — it validates frame *shape* and forwards opaque ciphertext; it holds no key and persists no plaintext, ciphertext, or frames. It optionally persists **room metadata only** (`{id, mode, expiresAt, backfillEnabled, createdAt, ephemeral}`, via `ROOM_DB_PATH`, default `:memory:`) to a SQLite file so room URLs survive a relay restart. The one piece of retained *content* state is the **Mode-A backfill buffer**: a bounded (`RECENT_CAP`) in-memory ring of recent ciphertext frames per room, replayed to late joiners and cleared the instant the room empties (never written to disk). It exists only for Mode A (where the relay provably can't decrypt), is opt-out per room at creation, and is forced off for **ephemeral rooms** (no device persists history — items auto-expire client-side; see `apps/web` below).

Packages (`packages/`) are consumed as **TypeScript source** (`main` → `src/index.ts`), not built artifacts — bundlers/vitest resolve `.ts` directly. The `build` task (`tsc -p`) emits in place and is essentially vestigial; those `.js`/`.d.ts` are gitignored under `packages/*/src`.

- **`protocol`** — Zod wire schemas; the single source of truth for frame shapes. `ClientFrameSchema` is a 13-member discriminated union: `clip` (type/msgId/iv/ciphertext/ts), `delete` (type/msgId — the msgId to remove; no plaintext), file transfer (`file-offer`/`file-accept`/`file-decline`/`file-chunk`/`file-ack`/`file-complete`/`file-cancel` — `file-offer` now carries only `{fileId,iv,ciphertext}`, its name/mime/size/hash sealed as ciphertext), and WebRTC signaling (`sdp`/`ice`/`rtc-hello`/`presence`). `ServerFrameSchema` adds `hello`/`peer-joined`/`peer-left`/`error`. `PROTOCOL_VERSION` (currently 2) rides on `hello.protocolVersion`, letting a client detect relay/peer skew. `CLOSE_CODES` (4404/4429/4413/4410 — the last is `ROOM_EXPIRED`), `MAX_FRAME_BYTES`. Client and relay both validate against these.
- **`crypto`** — AES-256-GCM envelope over WebCrypto. `deriveKey` (PBKDF2-SHA256, 200k iters) is **non-extractable by default** (pass `extractable: true` only in tests). Encryption is bound by AAD; a bounded `ReplaySet` dedups by msgId.
- **`room-code`** — pairing. `parseRoomUrl` is the shared URL contract: `/r/<routingId>#<secret>` → Mode A, `/r/<routingId>` (no fragment) → Mode B. `canonicalizeCode` (trim + uppercase) normalizes **custom Mode-B codes** so client and relay derive the identical key — in Mode B the code *is* key material, so canonicalization must be byte-identical on both sides; `isValidCustomCode` bounds length/charset.
- **`client-core`** — `UniclipClient`: derives the key, connects (`/ws/<routingId>` — **only the routingId is ever sent**), encrypts on send, and on receive runs shape→replay→decrypt before emitting. Exponential backoff reconnect, reset on the protocol `hello` (not the raw WS open — the relay accepts the upgrade and only afterward closes for a terminal condition); terminal close codes (`ROOM_NOT_FOUND` 4404 / `TOO_LARGE` 4413 / `ROOM_EXPIRED` 4410) stop auto-reconnect and surface an error, while `RATE_LIMIT` (4429) still reconnects. A `hello.protocolVersion` mismatch against `PROTOCOL_VERSION` emits an advisory `VERSION_MISMATCH` error once per instance, without blocking text sync. A **transport seam** (`sendFrame`) prefers an open WebRTC `RTCDataChannel` and falls back to the WS; `peer-link.ts` wraps `RTCPeerConnection` (injectable for Node tests) running perfect negotiation. Content (clip/delete/file-*) rides whichever pipe is live — `file-transfer.ts`'s `FileTransferManager` now encrypts the file-offer metadata (name/mime/size/chunkCount/hash/inline) under AAD `file-offer:${routingId}:${fileId}`, and the receiver re-validates the decrypted metadata against a Zod `OfferMetaSchema`, dropping it on failure; **signaling (`sdp`/`ice`) only ever rides the WS, in both directions**; a `transport: p2p|relay` event drives the Direct/Relayed UI badge. The app-layer AES-GCM envelope sits on top of DTLS, so content stays opaque on every path. `presence.ts` (`PresenceManager`) maintains the named device roster: it announces an **encrypted** `{id,name}` over the WS (AAD `presence:${routingId}`, relay-blind in Mode A), reconciles against `peerCount` with a TTL + fast-prune-on-leave, and emits a `presence` roster event. `rtc-hello` (P2P initiator handshake) and `presence` join `sdp`/`ice` as **WS-only** signaling (dropped if they arrive over the P2P pipe).
- **`apps/relay`** — Bun + Hono. `RoomStore` is a live cache over `room-db.ts`/`RoomDb` (a `bun:sqlite` store of room *metadata only*, the source of truth for room existence); `get()` rehydrates a room from the DB on a Map miss so URLs survive restart. `create()` forces `backfillEnabled = mode === "A" && backfill && !ephemeral`. GC: max-age closes every socket in the room with `ROOM_EXPIRED` (close 4410) before deleting from both DB+Map, idle evicts the Map only (rooms live to their 24h max-age); `gc()` returns `{aged, idle, expiredSockets}`, logged by the caller. Process-level `unhandledRejection`/`uncaughtException` handlers log and (for uncaught) exit so the container's healthcheck/restart policy recovers a clean process. Per-room `recent`/`backfillEnabled` for Mode-A backfill via `pushRecent`/`removeRecent`; per-room **tombstone set** (`tombstones: string[]`, bounded to `TOMBSTONE_CAP`) of deleted msgIds — replayed to (re)joiners via `addTombstone` so a peer offline at delete-time reconciles on reconnect; both `recent` and `tombstones` are cleared when sockets hit 0. `count` is live-Map size; `totalCount` (DB) backs `/api/health` + metrics. WS handlers (hello/peer-joined/peer-left/clip fan-out; delete frames are fanned out to all peers, the matching entry is pruned from the backfill ring via `removeRecent`, and the msgId is added to the tombstone set; on join replays `recent` then `tombstones` to the newcomer only; **WebRTC `sdp`/`ice`/`rtc-hello` signaling and encrypted `presence` frames are fanned out via the same `broadcast` and are never buffered, tombstoned, or persisted** — a dedicated `signalLimiter` rate bucket keeps signaling/presence off the clip limiter and is swept periodically alongside the others), per-socket + per-IP sliding-window rate limits (the per-IP key is the **last** `X-Forwarded-For` hop via `clientIp()` — a trusted-single-proxy assumption; the first hop is client-spoofable), Prometheus `/api/metrics` (incl. `uniclip_frames_dropped_total{reason}` and `uniclip_ws_closed_total{code}`), static SPA fallback (`STATIC_ROOT`), CORS on `/api/*`. **TURN minting:** `GET /api/ice` (`turn.ts`) mints short-lived coturn REST-scheme credentials (HMAC-SHA1 over a unix-expiry username) when `TURN_URLS`/`TURN_SECRET` are configured, else falls back to the static public-STUN `ICE_SERVERS`. **Update checker:** `GET /api/version` (`version.ts`) reports the running version/gitSha plus an hourly-cached, opt-out (`UPDATE_CHECK=off`) check against the latest GitHub release (`UPDATE_REPO`) so the UI can flag an available update. **Self-hosted CLI installer:** a dynamic `GET /setup.sh` route (`installer.ts` `renderSetupScript`) returns a POSIX installer templated with the request's base URL (`Host`+scheme, validated against an allowlist → 400 on a malformed Host) and the per-artifact SHA-256s read from `${STATIC_ROOT}/dl/checksums.txt`; the standalone CLI binaries are served as static files under `/dl/` (built into the image — see `apps/cli`).
- **`apps/web`** — Svelte 5 (runes) + Vite 6 + Tailwind 4. Path-based router (relies on the relay's SPA fallback for `/r/*`), `ClipboardWatcher` polling (`start()` probes clipboard readability once before arming the interval, so a permission denial surfaces to the caller instead of failing silently on every tick), encrypted-at-rest history in localStorage. The send composer is a single-line textarea (Enter to send, IME-safe via `composer-keys.ts`'s `isSendKey` — skips `isComposing` so a half-composed CJK string doesn't submit) with an expand-modal button; it enforces a `MAX_TEXT_BYTES` (32 KB, aligned to the protocol frame limit) cap — the Send button is disabled and a `{n} KB / 32 KB` counter appears when the limit is exceeded. Pairing is via `share-modal.svelte` (QR + copy-link of the room URL incl. the Mode-A `#secret`); a roster popover (`roster-popover.svelte`) shows connected devices by name, with a self-name editable inline (per-tab id in `sessionStorage`, name in `localStorage`, default from `defaultDeviceName()`). Items support **smart clips** (`clip-content.ts` linkifies http/https URLs → clickable `<a target=_blank rel=noopener>` + per-item Open/QR via `qr-popover.svelte`), **search/filter**, a **local pin** (`Item.pinned`, persisted; survives the 50-item cap — eviction drops oldest *unpinned* first via `persist.ts`'s `evictOldestUnpinned`, shared by both the live in-memory list and the persisted store so they apply the identical rule; never transmitted), and **export** (`export.ts`: copy-all + download .txt over the filtered items). **Ephemeral rooms** (`room.svelte`, driven by the `hello`/`room` event's `ephemeral` flag) swap `PersistedItems` for a no-op `EphemeralStore` and schedule each item for client-side auto-expiry — no device persists history.
- **`apps/cli`** — a Node + **Ink** (React/JSX, ESM, built with **tsup** → `npx`-able `uniclip` bin) terminal UI that creates/joins a **Mode-A** room and syncs **text and files**, reusing `client-core`. It handles the client's `delete` event (synced delete works in the TUI) and, for locally-sent items, keys the list entry off the **wire msgId** returned by `client.send()` rather than a locally-generated id, so a later delete/backfill reconciles correctly. Peer-controlled text (clip text, file-offer name, transfer name) is run through `sanitize-terminal.ts`'s `stripTerminal()` before render — it strips CSI/OSC/C0-C1 control sequences (an OSC-52 clipboard-hijack vector a TUI is exposed to that a browser isn't); this is display-only, copy-to-clipboard always uses the raw text. **Real P2P (P4b-i):** `werift-peer.ts` adapts the pure-TS **werift** WebRTC lib to the DOM `RTCPeerConnection` shape `PeerLink` consumes (Node has none natively), injected as `createConnection` so content travels peer-to-peer / LAN-direct with automatic relay fallback — `weriftPeer` is the default; `--relay-only` swaps in the never-opening `disabledPeer` stub to force relay. werift stays external to the tsup bundle; no persistence (in-memory session). The header's Direct/Relayed line is driven by the existing `transport` event. `uniclip` mints a room (`--relay`/`UNICLIP_RELAY`, default `http://localhost:3000`) + shows a UTF-8 QR; `uniclip <room-url>` joins. Keybindings via Ink `useInput` (↑↓ select, c/⏎ copy → OS clipboard via `clipboardy`, **f** send-file path prompt, q/Ctrl-C quit). `--version`/`--help` supported. Dev: `pnpm --filter @uniclip/cli dev`. **Standalone binary:** `src/bin.ts` is a dedicated `bun build --compile` entry (the tsup `cli.tsx` argv-guard stays false in a compiled binary, so bin.ts calls `main()` explicitly) — `scripts/build-binaries.sh` cross-compiles per OS/arch into `dist/dl/` + `checksums.txt`; the Docker `cli-builder` stage builds them into the image so the relay can serve `curl …/setup.sh` installs (no Node needed). Needs `react-devtools-core` as a devDep so Ink's dev-only import bundles; `--version` is the build-time-embedded `package.json` version (no runtime file read in a binary). **File transfer (Arc B):** reuses `client-core`'s `FileTransferManager` over the werift channel (chunking/AES-GCM/flow-control/hash unchanged) — `f` opens a path prompt (`readForSend` → `sendFile`); an incoming non-inline offer shows an accept/decline prompt; received files save to **`process.cwd()`** via `file-io.ts`'s `saveBlob`, which **sanitizes the peer-controlled filename** (basename only, strip leading dots/control chars — a path-traversal sink the web never had); `<Transfers>` shows progress. The file-prompt is a single `useInput` accumulator (no parallel `TextInput`, which would double-dispatch keystrokes). **Zero-internet (P4b-ii):** `uniclip --lan` is a fully offline host — it mints a Mode-A room locally, runs an **embedded `ws` fan-out relay** (`lan-relay.ts`, the relay's wire protocol minus backfill/tombstones/persistence — `hello`/`peer-joined`/`peer-left`/opaque-frame broadcast validated with `@uniclip/protocol`, with a `maxPeers` connection cap + per-socket sliding-window rate-limit using **independent post-validate budgets** — non-file 500/10 s, `file-*` 2000/10 s so a relay-fallback file transfer isn't throttled — as LAN DoS backstops), advertises it over **mDNS** (`mdns.ts`, `bonjour-service`, TXT carries `routingId` only — never the secret), points its own `UniclipClient` at `ws://127.0.0.1:<port>` (`iceServers: []`, host candidates, no STUN), and shows a QR `uniclip+lan://<routingId>#<secret>` pairing token (`lan-token.ts`). `uniclip <lan-token>` browses mDNS by routingId, resolves the host, and connects — syncing P2P over werift with no internet at all. `lan-session.ts` (`startLanHost`/`joinLan`, injectable `Discovery` for tests) wires it; `cli.tsx` routes `--lan`/token vs the relay path. CLI↔CLI only (a browser can't run an mDNS responder/local server, and LAN-HTTP isn't a secure context). Tests use `ink-testing-library` with an injected fake client and a fake `Discovery`, plus real two-peer **werift loopback** + **embedded-relay LAN e2e** tests in pure Node (no pty; `peerlink-werift.test.ts`/`lan-e2e.test.ts` deep-import/drive real `UniclipClient`s to prove a genuine handshake); the one real-multicast mDNS test is skip-guarded in CI.

### Security model (don't break these invariants)
- **Mode A is zero-knowledge**: the secret lives only in the URL fragment, is generated client-side, and is never put in any request — only `routingId` reaches the relay. Mode B derives the key from the routingId (which the server sees) and is labelled "less secure" in the UI.
- **AAD domain separation**: wire frames use `${routingId}:${msgId}` (file chunks: `${routingId}:${fileId}:${index}:${isFinal}`); at-rest persistence uses `persist:${roomId}`. **File-offer metadata is encrypted**: the `file-offer` frame carries only `{iv,ciphertext}` — name/mime/size/chunkCount/hash/inline are AES-GCM sealed under AAD `file-offer:${routingId}:${fileId}`, so the relay never sees a filename or hash; the receiver re-validates the decrypted metadata with a Zod schema and drops it on failure. Keep all these prefixes disjoint so a stored/other-purpose blob can't be replayed as a live frame.
- **Backfill is Mode-A-only by construction**: `RoomStore.create` forces `backfillEnabled = mode === "A" && backfill && !ephemeral`. Never buffer Mode-B clips — the relay can derive that key, so retaining Mode-B ciphertext would mean retaining readable history. Ephemeral rooms also never buffer (no device is meant to retain history for them at all). The buffer must stay memory-only and clear on empty (history lives only while a device is connected).
- **Persistence is metadata-only**: `RoomDb` (`bun:sqlite`) stores only `{id, mode, expiresAt, backfillEnabled, createdAt, ephemeral}` — never frames, keys, sockets, or the backfill ring. `ROOM_DB_PATH` defaults to `:memory:` (no cross-restart persistence unless configured). Adding any content column would break the zero-knowledge boundary.
- **Key derivation is single-sourced** in `client-core`'s `deriveRoomKey(room)` — both `UniclipClient` and `apps/web/src/routes/room.svelte` call it, so the Mode-A/B mapping has exactly one definition and stays identical across peers (the relay never derives keys). Don't reintroduce a second copy.
- **Synced delete is zero-knowledge**: a delete frame carries only the msgId (no plaintext or ciphertext); the relay fans it out and prunes the backfill ring without learning any content. The relay is authless, so any peer in the room can delete any item — this is acceptable for the trusted-room model.

## Toolchain gotchas (these will bite you)

- **Relay tests run under Bun, not Node** — and need **Bun ≥ 1.3** (1.1.x crashes vitest's tinypool worker teardown with "Cannot access 'dispose' before initialization"; CI pins 1.3.14). Its `test` script is `bun --bun vitest run`, and `apps/relay/vitest.config.ts` sets `server.deps.inline: [/zod/, /hono/, /ulid/]` (without it Bun's loader + vitest's transform produce two module copies and `z.object` is undefined) plus `pool: forks, singleFork: true` (one worker minimises the tinypool spawn/teardown surface). The integration tests need real `Bun.serve`. If a new relay test imports another CJS/ESM dep that comes back with undefined exports, add it to `inline`.
- **In `apps/relay/src/ws-handlers.ts`, mutate `raw.data.roomId` — never reassign `raw.data`.** Hono's bun adapter owns `ws.data` (`{events,url,protocol}`); reassigning it breaks `onClose`/`onMessage` dispatch.
- **`packages/tsconfig/node.json`'s base sets `"types": ["bun"]`** (the `@types/bun` reference name, not `"bun-types"`) directly — `apps/relay/tsconfig.json` extends it with no `types` override of its own, just `noEmit`/`resolveJsonModule`/`include: [src, test]`. `@types/bun` types `Response.json()` as `unknown`, so relay tests must cast: `(await res.json()) as {...}`.
- **`apps/web` needs `@sveltejs/vite-plugin-svelte` v5** (for Vite 6). v4 makes the dev server resolve Svelte's *server* build → `mount()` throws → blank page. The `vite build` still succeeds, so only the dev server / E2E catches it.
- **`apps/web` tests mock browser globals with `vi.stubGlobal`**, not `Object.assign(globalThis, …)` — `navigator` is a getter-only global in this Node and `Object.assign` throws.
- Helpers feeding WebCrypto return `Uint8Array<ArrayBuffer>` (not bare `Uint8Array`) so they satisfy `BufferSource` under TS 5.7's generic `Uint8Array`.
- **werift is patched (`patches/werift@0.23.0.patch`, via pnpm `patchedDependencies`).** Stock werift 0.23 falls back to a hardcoded public STUN (`stun.l.google.com`) whenever no STUN server is configured (`ice.js`: `this.stunServer = validateAddress(x) ?? [google]`), so `iceServers: []` would gather an srflx candidate — phoning home and **breaking `--lan`'s zero-internet guarantee** (and stalling ~5s on werift's hardcoded STUN timeout when offline). The patch drops that default so "no STUN configured" means **host candidates only**. If you bump werift, re-apply/re-verify the patch; the invariant is guarded by `apps/cli/src/werift-hostonly.test.ts` (real gather asserts host-only, no srflx). `pnpm install` applies the patch automatically; CI must run `pnpm install` (not a bare `node_modules` restore) so the patch lands. **`patchedDependencies` is declared in BOTH `pnpm-workspace.yaml` (pnpm ≥ 10's location) and `package.json`'s `pnpm` field (pnpm 9) — keep them in sync.** pnpm 10 ignores the package.json field (warns "keys were ignored") and pnpm 9 reads only it, so a machine on the "wrong" pnpm silently skips the patch and `--lan` leaks to Google STUN again. Watch for that WARN line in a `--verbose` run, or `srflx` candidates reappearing.

## Conventions

- TDD: write the failing test, confirm it fails, implement, confirm green. Commits are small and scoped (`feat(pkg): …`, `fix(pkg): …`).
- Trust `tsc --noEmit` / `svelte-check` / `vitest` exit codes over IDE diagnostics — stale "cannot find module" warnings are common right after creating a file; the file watcher lags writes.
- Spec and plan live in `docs/superpowers/{specs,plans}/`.
